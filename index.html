ROS-ros_control

Posted on 2018-08-02 | In ROS

1 ros_control概述

构建ros_control这样的robot-control框架的主要原因是在ROS中缺少实时安全通讯层，ros_control以完全robot-agnostic的方式关注实时性能（real-time performance）和控制器共用（sharing of controllers），帮助我们实现和管理机器人控制器。

ros_control的控制器生命周期（controller-lifecycle）、硬件资源管理（hardware resource management）和硬件接口抽象（abstractions on hardware interfaces）等实现方案尽可能少的考虑硬件和操作系统，使得ros_control框架更加通用化。

ros_control向MoveIt!与ROS Navigation stack开放标准的ROS接口，因此将支持ros_control的移动平台与机械臂结合在一起不需要编写额外的代码，只需配置controller文件即可。

ros_control提供一些库以支持实现自主设计的控制器。

2 ros_control repository

2.1 hardware_interface

ros_control框架的基础是硬件抽象层（Hardware Abstraction Layer），这是连接各种仿真和真实机器人的桥梁，相当于对我们要控制的机器人的抽象，或者说对被控机器人的模型描述。

硬件抽象层由hardware_interface::RobotHW类实现，具体的机器人实现都必须继承这个类。这个类的硬件实例模型由机器人提供，例如电动或液压执行器，编码器和力传感器等传感器。我们可以集成多种形式的硬件，仿真与真实机器人都可以。

一种可能的理想的构建机器人控制系统的方式是把已经实现的RobotHW实例组合到一起，各个部件来自于不同的供应商，每一个供应商都有自己特定的RobotHW实例。

hardware_interface包的其余部分定义了read-only或read-write类型的关节和执行器接口以对硬件进行抽象，例如state，position，velocity和effort接口等。这些接口是ros_control实现的标准化的接口，我们可以直接用来实例化RobotHW类，构建我们的机器人模型，当然，如果这些标准化接口不能满足要求时，我们也可以构建自己的抽象接口。

2.2 controller_manager

controller_manager负责管理控制器的生命周期，通过接口管理硬件资源，处理控制器间的资源冲突等。

controller的生命周期不是静态的，可以通过controller_manager提供的ROS Service在运行时进行查询和调整，这些服务允许我们在运行时启动、停止和配置控制器。

2.3 ros_control的其它package

ros_control还提供软件库处理实时ROS通信（real-time ROS communication），传动（transmissions）和关节限制（joint limits）。

realtime_tools包增加了类以realtime-safe的方式处理ROS通信。

transmission_interface包提供类实现joint-space和actuator-space的转换，例如：简单的减速器（simple reducer），四杆机构（four-bar linkage），差分传动（differential transmissions）等。ROS的机器人运动学、动力学描述文件URDF支持对transmission的描述。

joint_limits_interface包包含表示关节限制（joint limits）的数据结构，通过URDF或yaml文件描述它们的方法、执行这些限制的方法。

control_toolbox提供写controller时有用的组件：一个PID controller类，smoothers，正弦波和噪声生成器（sine-wave and noise generators）。

3 ros_controllers repository

ros_controllers仓库存放一些已有的controllers，支持绝大部分使用情况，例如机械臂、移动机器人、人形机器人等。

位置控制的机器人使用MoveIt!进行规划时，经常使用joint_trajectory_controller。

control_msgs提供ros_controllers中提供的大部分controllers所需的ROS messages。

Linux_udev规则编写及配置

Posted on 2018-06-21 | In Linux

最近在做移动机器人的开发，需要进行里程计（odometry）与IMU的数据融合，而里程计与IMU恰好都用到了同一个厂家的USB-Serial适配器，为了区分这两款传感器需要在原有的udev规则中添加一些新的的内容。

在命令行中可以查看USB设备信息：

pi@pi-desktop:~$ lsusb
Bus 001 Device 006: ID 10c4:ea60 Cygnal Integrated Products, Inc. CP210x UART Bridge / myAVR mySmartUSB light
Bus 001 Device 005: ID 1a86:7523 QinHeng Electronics HL-340 USB-Serial adapter
Bus 001 Device 004: ID 1a86:7523 QinHeng Electronics HL-340 USB-Serial adapter

原有的里程计udev规则如下：

ATTRS{idVendor}=="1a86", ATTRS{idProduct}=="7523", MODE:="0777", SYMLINK+="motor_trd"

原有的IMU udev规则如下：

ATTRS{idVendor}=="1a86", ATTRS{idProduct}=="7523", MODE:="0777", SYMLINK+="imu_driver"

可以看到有两个USB设备的idVendor与idProduct是一样的，因此如果按照原有的udev规则使用极有可能会发生冲突，导致设备无法正常使用。

解决方法是增加额外的信息来区分拥有相同idVendor与idProduct的设备。

IMU与里程计分别对应ttyUSB0与ttyUSB1。

在命令行中输入以下命令可以查看IMU对应的ttyUSB0的详细信息：

pi@pi-desktop:~$ udevadm info --attribute-walk --path=/sys/bus/usb-serial/devices/ttyUSB0

KERNEL=="ttyUSB*", KERNELS=="1-1.3", ATTRS{idVendor}=="1a86", ATTRS{idProduct}=="7523", MODE:="0777", SYMLINK+="motor_trd"

修改完成之后将.rules文件放入/etc/udev/rules.d文件夹中，像USB这类热插拔设备通常需要重新插拔才能使新规则生效。

MoveIt!学习笔记4-Kinematics and Collision Checking

Posted on 2018-04-03 | In MoveIt!学习笔记

Kinematics

MoveIt!使用插件基础设施，允许用户写自己的运动学算法。

Forward kinematics及jacobians已经集成到RobotState类中了。

默认的Inverse kinematics插件使用KDL numerical jacobian-based solver，这个插件通过MoveIt! Setup Assistant自动配置。

通常，用户可以选择实现自己的运动学解析器。一种常用的方法是用IKFast package生成机器人运行所需的C++代码。

Collision Checking

MoveIt!的碰撞检测用CollisionWorld对象配置在Planning Scene中，执行碰撞检测的库主要是FCL package。

MoveIt!支持不同类型对象的碰撞检测，包含：

  1. Meshes

  2. Primitive Shapes：例如boxes，cylinders，cones，spheres，planes

  3. Octomap：Octomap对象可以直接用于碰撞检测

Allowed Collision Matrix（ACM）

碰撞检测是一项非常昂贵的操作，通常会花费运动规划过程接近90%的算力。ACM编码二进制值，指示机器人本体中某两部分是否需要进行碰撞检测。如果某两部分在ACM中对应的值为1，说明这两部分不需要进行碰撞检测。例如，机器人本体中两部分可能相距特别远，不可能发生碰撞。

Time Parameterization

Motion Planner通常只会生成路径，并不会包含时间信息。MoveIt!包含一个trajectory processing routine，可以处理这些路径以生成服从关节最大速度与加速度约束，并进行time-parameterized的轨迹，其中速度与加速度限制从机器人的joint_limits.yaml文件中读取。

MoveIt!学习笔记3-Planning Scene

Posted on 2018-04-03 | In MoveIt!学习笔记

Planning Scene Monitor

Planning scene用于呈现机器人周围的世界及机器人自身的状态，它主要由MoveGroup节点中的planning scene monitor进行维护。

Planning scene monitor接收以下信息：

State Information：通过joint_states话题

Sensor Information：使用World Geometry Monitor

World geometry information：通过用户输入的planning_scene话题

World Geometry Monitor

World Geometry Monitor使用机器人传感器信息及用户输入构建world geometry。具体来说，World Geometry Monitor使用Occupancy Map Monitor对机器人周围环境进行3D呈现，并且通过planning_scene话题的信息来向场景中增加物体信息。

Occupancy Map Monitor（3D Perception）

MoveIt!通过Occupancy Map Monitor进行3D感知，Occupancy Map Monitor使用插件结构操控不同种类的传感器，MoveIt!内置支持两种类型的输入：

  1. Point clouds：由point cloud occupancy map updater插件处理

  2. Depth images：由depth image occupancy map updater插件处理

除此之外，我们可以以插件的形式增加自己的updater类型到Occupancy Map Monitor中。

Occupancy Map Monitor使用Octomap维护环境的occupancy map。Octomap可以对独立的单元编写概率信息，但MoveIt!当前没有使用这些信息。另外，Octomap可以直接传输到MoveIt!使用的碰撞探测库FCL。

Depth image occupancy map updater包含自己的self-filter，它会从深度图像中移走机器人的可视化部分，主要通过机器人当前状态信息执行该操作。

MoveIt!学习笔记2-Motion Planning

Posted on 2018-04-03 | In MoveIt!学习笔记

The Motion Planning Plugin

MoveIt!通过plugin接口使用不同的motion planners，这使得MoveIt!可以使用来自不同的库的motion planners。

MoveIt!到motion planners的接口通过MoveGroup节点提供的ROS Action和service实现。MoveGroup中默认的运动规划器使用OMPL进行配置，MoveIt!通过MoveIt!设置助手与OMPL进行交互。

The Motion Plan Request

Motion plan request清晰的指定我们希望motion planner做的事情。

通常，我们会请求motion planner将arm移动到不同的位置（在关节空间中）或将end-effector移动到新的pose。

碰撞探测（包含自碰撞探测self-collisions）默认执行。

我们可以将物体附着到末端执行器或机器人其它部位。例如，当机械臂抓取物体之后，我们可以将该物体附着到末端执行器，这使motion planner规划路径的时候考虑被抓取的物体的运动。

我们也可以指定希望motion planner进行检查的约束，MoveIt!提供的内置约束是kinematic约束：

  1. Position constraints - 限制某一link的位置在某一空间区域内

  2. Orientation constraints - 限制某一link的方向在指定的rpy角度范围内

  3. Visibility constraints - 限制某一link上的某一点在指定传感器的可见范围内

  4. Joint constraints - 限制关节角度位于指定的值之间

  5. User-specified constraints - 使用用户定义callback指定自己设定的约束

The Motion Plan Result

MoveGroup节点会生成要求的轨迹以响应motion plan request，轨迹会将arm移动到要求的位姿。

MoveGroup生成的是轨迹（Trajectory）而不仅仅是路径（Path），MoveGroup会使用要求的最大速度和加速度生成轨迹，轨迹在关节级别会服从速度与加速度约束。

The Motion Planning Pipeline:Motion planners and Plan Request Adapters

完整的motion planning pipeline chains将motion planner与planning request adapters组合到一起。Planning request adapters可以分别预处理motion plan requests及后置处理motion plan responses。

预处理在一些情形下比较有用，例如，机器人的起始状态稍微超出机器人指定的关节限制。

后置处理则在其它一些情形下比较有用，例如，将生成的路径转换为time-parameterized的轨迹。

MoveIt!提供一系列默认的motion planning adapters，每一种都执行特定的功能：

1 FixStartStateBounds

当实际机器人的关节限制没有进行合适的配置的情况下，机器人最终停到某一配置时，会有一个或多个关节稍微的超出关节限制，此时，motion planner无法进行规划，因为它认为机器人的起始状态超出了关节限制，这种情况下可以使用FixStartStateBounds适配器。

FixStartStateBounds适配器会移动Robot，使关节到达URDF文件指定的关节限制中。当然，这种方式不是在任何情况下都起作用的，例如，关节超出了关节限制很多的情形。适配器的参数可以指定关节超出关节限制多少的情况下是可被修复的。

2 FixWorkspaceBounds

FixWorkspaceBounds会指定默认的工作空间用于规划：10m10m10m的立方体。这个工作空间只有当motion planning request没有指定工作空间参数时才会指定。

3 FixStartStateCollision

FixStartStateCollision会通过微调关节值，在可能发生碰撞的指定的配置空间中采样获取新的无碰撞的配置。

通过“jiggle_factor”参数指定关节扰动量，控制扰动成为关节总的移动范围的某一百分比。该适配器的其它参数指定适配器在放弃之前进行多少随机扰动采样。

4 FixStartStatePathConstraints

FixStartStatePathConstraints在某一motion plan的起始状态不符合指定的路径约束时起作用。适配器会在机器人当前配置与服从路径约束的新位置之间规划一条新的路径。而新位置则作为规划的起始状态。

5 AddTimeParameterization

Motion Planner通常生成“kinematic paths”，例如，生成的路径并不服从任何的速度与加速度约束，而且并没有进行时间参数化。而AddTimeParameterization适配器则会构建服从速度与加速度约束且“time parameterize”的轨迹。

OMPL

OMPL（Open Motion Planning Library）是开源的运动规划库，主要实现随机化的运动规划器。MoveIt!直接集成OMPL并使用其中的运动规划器作为默认的规划器。OMPL中的规划器是抽象的，例如，OMPL并没有机器人的概念。而MoveIt!则将OMPL作为后端运行来解决机器人相关的问题。

MoveIt!学习笔记1-System Architecture

Posted on 2018-04-03 | In MoveIt!学习笔记

MoveGroup

MoveGroup是MoveIt!提供的主要节点，它将多个独立的组件组合到一起，提供我们常用的操作，如设置关节和位姿目标，创建运动规划路径，移动机器人，向环境增加对象，向机器人增加或移除对象等。

User Interface

MoveGroup提供一系列ROS actions，services，topics与User Interface进行交互。用户可以通过C++实现的move_group_interface，python实现的moveit_commander及RViz的MoveIt!插件Motion Planning与MoveGroup进行交互。

ROS Param Server

ROS Param Server则向MoveGroup提供机器人的URDF，SRDF及其它配置参数。配置参数包括关节限制，运动学参数，运动规划参数，感知及其它信息等，配置文件由MoveIt! setup assistant生成并放在config文件夹中。

Robot Interface

MoveGroup通过ROS topics及actions与Robot进行交互，获取机器人当前状态信息（如关节位置），点云信息及其它传感器信息，并与机器人Controller进行交互。

Joint State Information

MoveGroup通过接收/joint_states topic确定机器人每个关节位置，MoveGroup可以同时接收多个publishers发布的/joint_states topic，即使这些publishers发布的只是局部的机器人状态信息，如两个publishers分别发布arm与移动底盘的状态信息。

关节状态publisher是在Robot侧实现的，MoveGroup并不构建joint state publisher。

Transform Information

MoveGroup使用ROS TF库监测变换信息，以此来获取机器人pose的全局信息（例如，ROS navigation stack会将map frame与base frame之间的变换发布到TF）。

MoveGroup只接收TF信息，因此我们需要在Robot侧构建robot_state_publisher节点发布TF信息。

Controller Interface

MoveGroup使用FollowJointTrajectoryAction接口与Robot controller进行交互。Robot侧需要设置FollowJointTrajectoryAction的server，MoveGroup则会实例化client与server进行交互。

Planning Scene

MoveGroup使用Planning Scene Monitor维护planning scene，planning scene是对机器人周边环境及当前机器人状态的呈现，其中机器人状态包含机器人抓取的物体。

Extensible Capabilities

MoveGroup容易进行扩展，像抓取放置功能、运动学、运动规划都通过插件的形式实现，这些插件通过ROS yaml参数进行配置，并使用了ROS pluginlib库。通常我们不需要手动对这些插件进行配置，使用MoveIt! Setup Assistant构建MoveIt! package的时候会自动配置。

ROS学习笔记8-Catkin编译系统

Posted on 2018-04-02 | In ROS学习笔记

Linux下的编译器有gcc、g++，随着源文件的增加，直接用gcc/g++命令的方式显得效率低下，大家开始用Makefile进行编译。然而，随着工程体量的增大，Makefile也不能满足要求，于是就出现了CMake工具。

CMake是对make工具的生成器，是更高层的工具，它简化了编译构建过程，能够管理大型项目，具有良好的扩展性。ROS采用CMake，并对CMake进行了扩展，形成了Catkin编译系统。

1 Catkin软件包（package）的特点

一个Catkin软件包（package）必须要包括两个文件：

  1. package.xml：包含了package的描述信息
  2. CMakeLists.txt：构建package所需的CMake文件，调用Catkin的函数/宏，解析package.xml，找到其他依赖的catkin软件包，将本软件包添加到环境变量

2 Catkin工作原理

Catkin编译的工作流程如下：

  1. 首先在工作空间catkin_ws/src/下递归的查找其中每一个ROS的package.
  2. package中会有package.xml和CMakeLists.txt文件，Catkin编译系统依据CMakeLists.txt文件，从而生成makefiles（放在catkin_ws/build/中）。
  3. 然后make刚刚生成的makefiles等文件，编译链接生成可执行文件（放在catkin_ws/devel中）。
    可以看出，Catkin就是将cmake与make指令做了一个封装从而完成整个编译过程的工具。

3 CMakeLists.txt文件解析

CMakeLists.txt原本是CMake编译系统的规则文件，而Catkin编译系统基本沿用了CMake的编译风格，只是针对ROS工程添加了一些宏定义。所以在写法上，catkin的CMakeLists.txt与CMake的基本一致。

这个文件直接规定了这个package要依赖哪些package，要编译生成哪些目标，如何编译等等流程。它指定了由源码到目标文件的规则，catkin编译系统在工作时首先会找到每个package下的CMakeLists.txt，然后按照规则来编译构建。

通过rosbag，能够将发布在一个多个话题上的消息录制到一个包文件（bag file）中，然后可以回放这些消息，重现相似的运行过程，这会极大的提高我们测试机器人算法的便利性。

包文件（bag file）是指用于存储带时间戳的ROS消息的特殊格式文件，rosbag命令行工具可以用来录制和回放包文件。

2 回放包文件

存储在包文件中个的消息将被回放，而且回放时会保持与其原始发布时同样的顺序和时间间隔。

录制与回放注意事项:

  1. rosbag会创建新的节点，该节点订阅指定的话题来记录消息，并且使用匿名机制，因此我们可以同时运行多个rosbag创建的节点实例。

  2. 同样的，在回放bag文件时，rosbag同样会创建新的节点发布bag文件中存储的消息。

  3. rosbag不会记录服务调用内容，只记录发布到话题的消息。
