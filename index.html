ROS学习笔记4_日志系统

Posted on 2018-07-29

1 日志的生成

1.ROS_INFO_STREAM(message)这样的宏可生成日志消息流，其中，message可以处理C++中标准输出流（ostream）中的各种表达式，比如std::cout，在一些数据类型上使用“<<”操作符等；也可以处理标准流操作符，如std::fixed，std::setprecision，std::boolalpha等。
需要注意的是，这些标准流操作符的作用范围只限于其出现的日志消息中，在其他日志消息中，任何想要使用的操作符必须重新插入。原因是ROS_INFO_STREAM这样的宏展开对应的短小代码块构造了一个标准字符串流（std::stringstream）对象，并将我们提供的参数插入这个字符串流；然后，这些代码将字符串流找那个完全格式化的内容发送至一个内部的日志系统，这个过程完成后，标准字符串流对象将被销毁，它的内部状态都将丢失。
另外，message中也没必要使用std::endl等行终止符，因为日志系统本身是面向行的，调用这些宏中任意一个将生成显示为一行的一个完整的日志消息。

  1. ROS_INFO_STREAM宏提供了C++风格的输出流接口，ROS_INFO宏提供格式输出函数printf风格的接口。

  2. ROS_INFO_STREAM_ONCE宏内部使用static变量确保宏在第一次被调用的时候生成消息，后续再次被调用时则不产生新的消息。比如在循环中使用这个宏，那么只在第一个循环产生日志消息，后续循环不再产生。

  3. ROS_INFO_STREAM_THROTTLE(interval,message)宏通过interval指定相邻日志消息出现的最小时间间隔，第一次执行时会生成日志消息，随后必须经过大于指定的时间间隔的时间才会生成新的日志消息。

2 日志的查看

通过ROSCONSOLE_FORMAT环境变量来调整日志消息打印到控制台的格式，默认格式是：

[${severity}][${time}]:${message}

除了默认的域，还可选择其他的域，如用${file}、${line}、${function}查看生成日志消息的源代码位置；用${node}域查看生成日志的节点名称等。

roslaunch工具默认不会将标准输出和标准错误从其生成的节点导入至自己的输出流，为了查看节点的输出，必须显式的使用output=”screen”属性，或在命令行中roslaunch命令后使用“–screen”参数。

/rosout节点通过/rosout话题将系统中所有节点产生的日志汇总，通过/rosout_agg话题进行发布，通常/rosout节点作为系统中唯一的日志消息订阅节点，这样可以避免系统中很多节点同时订阅/rosout话题时产生的大的系统开销。调试工具通过订阅汇总之后的/rosout_agg话题就可以获得完整的日志消息流。

/rosout节点还会生成记录日志消息的日志文件，文件名类似于：

~/.ros/log/run_id/rosout.log

运行标识码run_id是一个通用唯一识别码（UUID），Master运行时基于计算机的MAC地址和当前的时间生成。

rosclean check指令查看被ROS日志消耗的硬盘空间，rosclean purge删除所有已经存在的ROS日志。

3 日志级别的调整

每个节点都会提供set_logger_level服务，通过调用该服务可以调整节点产生日志的级别，ROS的C++程序默认只生成INFO或更高级别的消息。

ROS程序内容可通过调用ROS实现日志功能的log4cxx提供的接口调整节点的日志级别。

ROS学习笔记3_命名系统

Posted on 2018-07-29

1 全局名称

全局名称在任何地方都有明确的含义，无需额外的上下文信息（例如节点的命名空间等）来决定名称指的是哪个资源。
全局名称通常由前斜杠“/”，由斜杠分开的一系列命名空间，描述资源本身的基本名称构成。

2 相对名称

相对名称使用ROS为其提供的默认命名空间，典型特征是缺少全局名称带有的前斜杠。如果不知道相应的默认命名空间，那么相对名称不能和特定的计算资源匹配。
默认命名空间是单独地为每个节点设置的，而不是在系统范围进行。ROS默认会将全局命名空间（/）作为一个节点的默认命名空间，改变默认命名空间的最好和最常用的方法是在launch文件中使用命名空间（ns）属性：__ns:=default-namespace或ns=”default-namespace”。

3 私有名称

私有名称使用它们的节点名称作为命名空间而不是当前默认命名空间，这是它与相对名称的区别。使用这种名称的主要目的是，有一些节点内部的资源只与本节点相关，不会与其他节点打交道，用私有名称为这些节点命名，通常是节点使用的参数，话题（Topic）通常不能以私有名称命名。

4 节点匿名机制

将ros::init_options::Anonymous-Name作为第四个参数传递给ros::init()，将为节点设置匿名名称，匿名不意味着节点没有名字，而是使用处理器时间（wall clock time）在节点基本名称之后追加某个额外的文本，以确保节点的名字是唯一的。因此，我们可以启动同一节点的大量副本，而不导致命名冲突。

Linux_udev规则编写及配置

Posted on 2018-06-21

最近在做移动机器人的开发，需要进行里程计（odometry）与IMU的数据融合，而里程计与IMU恰好都用到了同一个厂家的USB-Serial适配器，为了区分这两款传感器需要在原有的udev规则中添加一些新的的内容。

在命令行中可以查看USB设备信息：

pi@pi-desktop:~$ lsusb
Bus 001 Device 006: ID 10c4:ea60 Cygnal Integrated Products, Inc. CP210x UART Bridge / myAVR mySmartUSB light
Bus 001 Device 005: ID 1a86:7523 QinHeng Electronics HL-340 USB-Serial adapter
Bus 001 Device 004: ID 1a86:7523 QinHeng Electronics HL-340 USB-Serial adapter
Bus 001 Device 003: ID 0424:ec00 Standard Microsystems Corp. SMSC9512/9514 Fast Ethernet Adapter
Bus 001 Device 002: ID 0424:9514 Standard Microsystems Corp. SMC9514 Hub
Bus 001 Device 001: ID 1d6b:0002 Linux Foundation 2.0 root hub

原有的里程计udev规则如下：

ATTRS{idVendor}=="1a86", ATTRS{idProduct}=="7523", MODE:="0777", SYMLINK+="motor_trd"

原有的IMU udev规则如下：

ATTRS{idVendor}=="1a86", ATTRS{idProduct}=="7523", MODE:="0777", SYMLINK+="imu_driver"

可以看到有两个USB设备的idVendor与idProduct是一样的，因此如果按照原有的udev规则使用极有可能会发生冲突，导致设备无法正常使用。

解决方法是增加额外的信息来区分拥有相同idVendor与idProduct的设备。

IMU与里程计分别对应ttyUSB0与ttyUSB1。

可以看到第三层级的设备描述信息中的KERNELS属性是不同的，因此可以用该属性对设备进行区分。

修改后的IMU udev规则如下：

KERNEL=="ttyUSB*", KERNELS=="1-1.2", ATTRS{idVendor}=="1a86", ATTRS{idProduct}=="7523", MODE:="0777", SYMLINK+="imu_driver"

修改后的里程计udev规则如下：

KERNEL=="ttyUSB*", KERNELS=="1-1.3", ATTRS{idVendor}=="1a86", ATTRS{idProduct}=="7523", MODE:="0777", SYMLINK+="motor_trd"

修改完成之后将.rules文件放入/etc/udev/rules.d文件夹中，像USB这类热插拔设备通常需要重新插拔才能使新规则生效。
